# assets/

Static assets bundled into the Flutter app at build time.

---

## data/ps/

Only `formats.json` here is current — 32 manually-curated competitive format definitions loaded by `FormatService` (`lib/services/format/`) at startup. It isn't PS-sourced data, so `scripts/sync_ps_data.py` doesn't touch it; edit it by hand.

Every other file in this directory (`learnsets.json`, `event_learnsets.json`, `moves.json`, `items.json`, `abilities.json`, `learnsets-g6-allowlist.json`, `version.json`) is a **frozen pre-migration snapshot** — `sync_ps_data.py` used to write here, but now writes exclusively to [`shared/ps_data/`](#sharedps_data-project-root) (see below). Nothing reads these stale copies anymore; don't regenerate or rely on them.

---

## shared/ps_data/ (project root)

The current, live-updated Pokémon Showdown data — generated by `scripts/sync_ps_data.py`, bundled into the Flutter asset tree via `pubspec.yaml`'s `shared/ps_data/` entry, and read directly by the backend (`PS_DATA_DIR` env var, default `../shared/ps_data` locally / `/app/shared/ps_data` in Docker). One copy, one source of truth for both sides — no more `assets/data/ps/` + `backend/app/static/` duplication.

| File | Contents |
|------|----------|
| `learnset_1.json` … `learnset_9.json` | Per-generation, per-Pokémon move lists (method, level, `via_prevo`), derived from PS's TypeScript learnset source |
| `pokedex.json` | Species/variety base data |
| `pokedex-gen-overrides.json` | Per-gen stat/type/ability overrides for Pokémon that differ from their Gen 9 defaults |
| `moves.json` | Move stats: type, power, accuracy, PP, Z-move and Max move data |
| `items.json` | Item data: generation introduced, Mega stone → form, Z-crystal → move mappings |
| `abilities.json` | Ability data: generation introduced |
| `formats-data.json` | Per-format tier/banlist data |
| `version.json` | SHA-256 hash of each file above; compared by `FormatService` (Flutter) and served via `GET /ps-data/version` (backend) to detect updates |

`FormatService` loads `assets/data/ps/formats.json` plus these files at startup (from Hive cache if available, otherwise from the bundled assets), then checks `GET /ps-data/version` in the background. If any SHA differs, it downloads the updated file from `GET /ps-data/file/:name` and saves it to Hive. `CatalogService` and `PokemonResolverService` on the backend read the same directory directly off disk — no download step needed server-side.

To update the PS data:

```bash
pip install requests json5
python scripts/sync_ps_data.py
git add shared/ps_data/
git commit -m "chore: update PS data"
# Redeploy backend — clients detect the SHA change and auto-download
```

## shared/pokemon_registry.json (project root)

Consolidated override maps consumed by `PokemonDataRegistry` (`lib/data/pokemon_data_registry.dart`) on app startup — cosmetic form labels/sprite URLs, base form name overrides, mega stone mapping, version-group lookups, regional form lookup, and `varietyIconIdOverrides` (icon resolution for specific Pokémon varieties). Replaces what used to be hand-maintained Dart const maps spread across half a dozen files. The backend's aggregation endpoints (`PokemonResolverService`, `LearnsetService`) read this same file directly from `shared/` — it's one copy, not a pair to keep in sync. (`assets/data/pokemon_registry.json` is a stale pre-migration leftover; nothing reads it.)
